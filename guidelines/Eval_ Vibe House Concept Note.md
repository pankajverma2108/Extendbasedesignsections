## **4\. Technical Specifications & Error Handling**

### **4.1 Match Logic (Vision AI)**

* **Face Match:** Required Score \> 0.85. If \< 0.85, allow 1 retry, then flag for manual override.  
* **Document Comparison:** System compares the OCR text of the Onsite ID Scan with the Remote ID Scan. If the ID Number or Name differs, the flow is blocked.

### **4.3 Edge Case Handlers**

* **Input Blur:** If Onsite ID Scan is blurry, the "Submit" button remains disabled until a sharp frame is detected.  
* **Smart Lock Offline:** If the Lock API fails, display "Staff notified. Please collect a physical key" and send a P1 alert to the Manager.

## **5\. Compliance Checklist**

* **Consent:** Guest must click a mandatory checkbox: *"I consent to the digital processing of my ID for check-in purposes"* during the Pre-checkin phase.

## **6\. Hardware Specifications**

To ensure smooth AI processing (Face Match) and clear document capture, the following minimum specifications are mandated for property rollout:

* **Device:** 11-inch or 12.9-inch Tablet (iPad Air/Pro or Samsung Galaxy Tab S9).  
* **Camera:** Minimum 12MP Front Camera with Wide-Angle support (for selfies) and 12MP Rear Camera (for document scanning).  
* **Processor:** Apple M-series or Snapdragon 8 Gen 2 (required for local liveness processing and fast OCR).  
* **Mounting:** Desktop or Floor-stand enclosure with integrated charging and a physical lock to prevent port tampering.  
* **Connectivity:** Wi-Fi 6 \+ 4G/5G Failover (SIM card slot required for properties with unstable Wi-Fi).

## **7\. Kiosk Environment & Deployment**

The tablet must run in a "Locked Kiosk Mode" to prevent guests from exiting the app or changing system settings.

### **7.1 Managed Mode (MDM)**

* **Tool:** Use an MDM (Mobile Device Management) solution like **Microsoft Intune**, **Jamf**, or **Google Workspace Endpoint Management**.  
* **Configuration:**  
  * **Single App Mode:** The tablet is locked to the Sarathi Web App URL.  
  * **Disable Notifications:** All system alerts, OS updates, and pop-ups are suppressed.  
  * **Power Management:** Set to "Never Sleep" while connected to power; auto-dim during property quiet hours (e.g., 2 AM \- 6 AM).

### **7.2 Progressive Web App (PWA) Architecture**

The application will be deployed as a PWA to ensure:

* **Full-screen UI:** Removes browser address bars and navigation buttons.  
* **Service Workers:** Basic assets (CSS, JS, Logos) are cached locally so the UI loads instantly even on slow connections.  
* **Hardware Access:** Explicit permissions for Camera and GPS are granted at the MDM level so guests aren't prompted.

### **7.3 On-Device Processing vs. Cloud**

* **Liveness Detection:** Performed on-device using a lightweight TensorFlow.js model to detect eye-blinks or head movement. This reduces latency and bandwidth.  
* **Face Match & OCR:** The high-res frames are sent to **Vision API** via Firebase Functions.  
* **Image Comparison:** The "Onsite Scan" is compared against the "Remote Upload" using a Structural Similarity Index (SSIM) algorithm to detect if the physical document is a photocopy or the original.

## **7.4. Security & Compliance**

* **Session Purge:** Upon "Check-in Complete" or "Session Timeout," all local blobs/caches of guest photos are wiped from the tablet's memory.  
* **Encrypted Transit:** All data sent via HTTPS (TLS 1.3).  
* **Physical Security:** The stand must hide the "Home" button and block USB ports to prevent unauthorized data extraction.

## **8\. Operational Maintenance (Ops Team)**

### **8.1 Monitoring by Property manager:**

* Real-time status of all tablets (Online/Offline, Battery Level).  
* **Success Rate:** Percentage of guests completing check-in without staff help.

## **3\. Product Catalog & Pricing Logic**

The following catalog is standard for the Upsell Engine:

* **Commodities:** Water Bottle (₹100), Bath Towel (₹200), Safe Lock (₹150), Toilet Kit (₹150) .  
* **Services:** Laundry (₹150), Early Check-in (₹250).  
* **Dynamic Pricing Engine:** Late Checkout is priced at ₹250 during the pre-arrival phase. The system must enforce logic where same-day late checkout requests are priced higher than this pre-booked rate.

## 

## **4\. Technical Specifications & Error Handling**

### **4.1 Integration Architecture**

* **Inventory Master:** eZee PMS tracks maximum available physical commodities (e.g., total safe locks).  
* **Payment Master:** Vibe House custom backend via Razorpay. All small-item transactions strictly enforce a prepaid, instant payment model (no "pay-later" options) .  
* **Folio Sync:** Once Razorpay confirms the payment, the Vibe House backend makes a POST request to the eZee PMS API to append the items to the guest's invoice and update the ledger to reflect the external payment.

### **4.2 Edge Case Handlers**

* **Payment Drop-off / Failure:** If Razorpay returns a FAILED status or the user abandons the cart, the temporary inventory hold is released after 10 minutes. The PMS folio is *not* updated.  
* **PMS Sync Failure:** If the payment succeeds on Razorpay but the eZee PMS API is unresponsive, log the transaction in the Vibe House DB as SYNC\_PENDING. A cron job will retry the API call every 5 minutes until a 200 OK response is received from eZee.  
* **Inventory Conflict:** If an item goes out of stock between cart addition and payment, Razorpay initialization is blocked, and the user is prompted with an "Item Unavailable" toast notification.

## **5\. Security & Compliance**

* **Payment Data:** Vibe House backend does not store any sensitive card or UPI data; tokenization and processing are fully offloaded to Razorpay.  
* **Invoice Integrity:** To prevent revenue leakage, operational tasks (like handing over a towel or assigning early check-in) are strictly gated and cannot be generated for staff until the payment webhook validates the transaction.

## **2\. Task Lifecycle & Statuses**

Every service request generated by the system (whether triggered by a guest's payment or a free request) moves through three distinct phases:

2. **In-Progress:** The assigned staff member has clicked "Acknowledge/Pick," and the system updates the database. The staff is currently executing the task.

## **3\. Auto-Assignment Logic (Load Balancing)**

To prevent overwhelming any single staff member, the system uses a dynamic load-balancing algorithm.

* **Trigger:** A new service request is generated (e.g., "Deliver Bath Towel to Room 101").  
* **Staff Query:** The system identifies all staff currently "Active/On-Shift" for that specific department (e.g., Housekeeping).  
* **Load Calculation:** The system counts how many tasks currently have the status In-Progress for each active staff member.  
* **Assignment:** The task is automatically assigned to the staff member with the minimum (min) number of In-Progress tasks.

## **4\. SLA & Escalation Protocol**

The system enforces a strict acknowledgment and escalation loop to ensure no task is ignored. Based on your escalation matrix, the loop runs sequentially through predefined levels (e.g., $L\_0$: Assigned Staff, $L\_1$: Reception, $L\_2$: Manager, $L\_3$: Director/Owner, $L\_4$: Tech).\\

![][image20]

**The Escalation Loop ($L\_x$ to $L\_{x+1}$):**

1. **Initial Notification:** Level $L\_x$ is notified of the task and must click the "Acknowledge" button.  
2. **Timer 1:** The system waits for $X$ minutes.  
3. **DB Check 1:** The system checks the DB to see if the "Acknowledge" button was clicked.  
   * *If Acknowledged:* The DB is updated to In-Progress, and no further escalation occurs.  
   * *If Not Acknowledged:* A "Reminder" notification is sent to $L\_x$.  
4. **Timer 2:** The system waits for another $X$ minutes.  
5. **DB Check 2:** The system checks the DB again.  
   * *If Acknowledged:* No further escalation occurs.  
   * *If Not Acknowledged:* The task automatically escalates to $L\_{x+1}$ (the next level up), and the exact same notification loop restarts for the higher authority.

Staff members may encounter legitimate delays (e.g., a room is locked, or a spare part is needed) that prevent them from completing an In-Progress task within the allotted time.

* **Request:** The staff member clicks "Request SLA Pause" on their interface and selects a reason from a dropdown.  
* **Approval Flow:** A notification is instantly sent to their upper-level supervisor (e.g., Manager).  
* **Action:** The Manager must review and click "Approve" or "Reject".  
  * *Approved:* The SLA timer freezes, preventing an artificial breach/escalation.  
  * *Rejected:* The SLA timer continues running, and the staff must complete the task immediately.

# **Vibe House: eZee PMS API Requirements Document**

**Project:** Custom PWA (Guest Interface) & Automation Engine 

**Objective:** To integrate Vibe House's custom middleware with eZee PMS to automate web check-ins, add-on sales, and dynamic inventory management.

Please provide the corresponding API Endpoints, HTTP Methods, and sample JSON payloads (Request/Response) for the following functional requirements:

### **1\. Listing & Availability: Unit-Level (Room or Bed)**

**Context:** Vibe House operates a hybrid model with both private rooms and dormitories (1 room, *n* beds). The custom engine will dynamically handle stay extensions, availability syncs, and bed-level upgrades. **Required APIs (CRUD Operations):**

* **Create/Update Unit:** APIs to add or modify individual physical beds or rooms within the PMS.  
* **Read (Live Availability Sync):** Fetch real-time availability of specific rooms and individual beds to support the dynamic pricing engine and automated extension requests.  
* **Update (Bed Assignment/Modification):** Retrieve the specific bed number assigned to a guest within a dorm, and update this assignment (e.g., if the guest purchases a "Lower Bunk Upgrade").  
* **Delete/Deactivate:** Temporarily block or deactivate a bed/room for maintenance.

### **2\. Reservations & Bookings (with Payments)**

**Context:** The middleware will handle direct bookings and modify existing OTA bookings, along with syncing payment statuses to the financial folio. **Required APIs (CRUD Operations):**

* **Create Booking (Single/Group):** Push a new reservation into eZee for either a single bed/room or a group of beds, along with the captured payment details.  
* **Read Booking Details (Single/Group):** Retrieve a reservation using the Booking ID or OTA reference number, returning guest names, dates, room category, assigned bed/room, and linked group IDs.  
* **Update Booking & Profile:** Push guest KYC details (email, phone, address, and ID verification status) back into the reservation record. Update stay dates for automated extensions.  
* **Update Payment Status:** API to record a payment gateway transaction as 'Paid' against the specific reservation's folio (Single or Group folio).  
* **Delete/Cancel Booking:** Cancel a reservation and update the available inventory.

### **3\. Check-ins & Check-outs**

**Context:** Guests will use the custom web app to complete pre-arrival registration and check themselves in/out. Group members may arrive at different times. **Required APIs:**

* **Single Check-in / Check-out:** Change a standard reservation status from "Reserved" to "Checked-In" or "Checked-Out".  
* **Group Check-in / Check-out:** Bulk update the status for an entire linked group booking simultaneously.  
* **Partial (Split) Check-in:** Ability to mark specific beds/guests within a single group booking ID as "Checked-In" while leaving the remaining group members' statuses as pending.

### **4\. Services, Commodities & Inventory**

**Context:** Guests will purchase physical items (water bottles, towels) and services (early check-in, late checkout) via the custom PWA. Payment is processed externally by Vibe House, but the transactions must be synced. **Required APIs (CRUD Operations):**

* **Read (Service/Commodity Listing):** Fetch Catalog / Master Data to retrieve the list of active commodities and services with their base prices from eZee.  
* **Create/Book Service with Payment (Single/Group):** \* Post Extra Charge (Services): Add a financial line item to the folio for time-based services like Early Check-in.  
  * Post Commodity Transaction: Push a completed order for physical inventory to eZee, recording the item on the invoice and marking it as 'Paid'. Must specify if the charge applies to a Single folio or is split/added to a Group folio.  
* **CRUD Inventory:** Operations to create new sellable inventory items, read current stock levels, update stock counts (deduct upon purchase), and delete/retire items.

**Project:** Custom PWA (Guest Interface) & Automation Engine 

**Objective:** To integrate Vibe House's middleware with MyGate's smart lock ecosystem to completely automate property management, manager assignments, and guest access (PIN generation/revocation).

**Context:** Our backend server needs to authenticate automatically as a "Partner Admin" , and we need to facilitate manager logins. Required APIs:

* Admin/Manager Login: Endpoint to authenticate a manager or Partner Admin programmatically (e.g., bypassing the manual mobile OTP login if possible, via API Key or OAuth2 token for server-to-server calls) .  
* Session Management: APIs to refresh tokens and maintain an active session.

### **2\. Property & Room Management (CRUD Properties/Hotels)**

**Context:** We need to programmatically mirror our eZee PMS property and room structures within the MyGate system. Required APIs:

* Create Property/Room: Endpoints to add a new building/property and map specific rooms to it .  
* Read Properties/Rooms: Fetch the list of all properties and their associated rooms, including the serial numbers and setup status of the connected locks .  
* Update Property/Room: Modify property details or room names.  
* Delete Property/Room: Remove a room or property from the system.

### **3\. Manager Assignment & App Access (CRUD Managers)**

**Context:** We need to provision, update, and revoke access for our operations staff and building managers dynamically. Required APIs:

* Create Manager/App Access: Add a new manager to the system and grant them "App Access" for specific properties/rooms . This includes toggling the "Make authorised admin" parameter.  
* Read Managers: Retrieve a list of all assigned building managers and their current access levels .  
* Update Manager: Modify a manager's access scope (e.g., assigning them to a new property or changing their admin privileges).  
* Delete/Revoke Manager Access: Instantly remove a manager's access upon shift end or termination.

### **4\. Access Control (CRUD PIN Codes)**

**Context:** When a guest completes web check-in, our system must instantly generate a lock PIN. *Note regarding modifications: We do not require an API to update, prolong, or extend an active PIN. * Create PIN: Programmatically generate a PIN matching the dashboard's capabilities . Required parameters include:

  * `PIN_Type`: Specify either "Auto PIN" or "Custom PIN" .  
  * `Validity_Period`: Must support both "Timed" (for guests, requiring specific start/end date-times) and "Permanent" configurations .  
* Read PINs: Fetch all active and scheduled PINs for a specific room to monitor capacity and group access .  
* Delete/Revoke PIN: Ability to immediately delete or disable a PIN (e.g., upon early checkout, cancellation, or when generating a new PIN for a late checkout).

### **5\. Telemetry, Monitoring & Webhooks**

**Context:** To prevent operational failures and track actual guest arrivals. Required APIs / Webhooks:

* Access Logs: Endpoint to fetch (or webhook to receive) real-time unlock events. We need to know when a specific PIN was used to verify actual guest arrival.

**Vibe House System Architecture Proposal** 

**1\. Load & Scale Assessment**

Expected Load: 500 Requests Per Second (RPS) ≈ 30,000 Requests Per Minute (RPM).

* Verdict: This is a moderate-to-high scale system. It requires careful database connection pooling, robust caching, and horizontal scalability to prevent bottlenecks, especially during peak check-in times or social events.  
* Primary Design Goal: High Availability (HA), Zero-Downtime Deployments, and strict Rate Limiting to protect external APIs (eZee and MyGate) which might not natively handle 500 RPS.

**2\. Platform Architecture**

**A. Frontend (Guest Interface PWA & Staff Dashboard)**

* Tech Stack: Next.js (React) or Vite (React/Vue).  
* Hosting: Vercel or AWS CloudFront/S3.  
* Why: Edge caching is mandatory at 500 RPS. Static assets and the PWA shell must be served from CDN edge nodes, never hitting the backend servers.

**B. Backend Middleware (The "Brain")**

* Tech Stack: Node.js (Express / Fastify) or Go (Golang). Node.js with Fastify is excellent for high JSON throughput, but Go is superior for raw concurrency and low memory usage at 500 RPS.  
* Hosting & Deployment: Containerized in Docker, orchestrated via AWS ECS (Fargate) or Kubernetes (EKS).  
* Zero-Downtime Deployment (CRITICAL):  
  * Must use Blue/Green Deployments or Rolling Updates.  
  * The Load Balancer (AWS ALB) slowly shifts traffic to new containers only after they pass strict health checks, draining old connections gracefully so no guest API requests drop during a deployment.

**C. Databases & Caching**

* Primary Relational DB (SQL): PostgreSQL (AWS Aurora Serverless v2 or RDS).  
  * Handles ACID transactions (Payments, Audits).  
  * Critical Addition for 500 RPS: PgBouncer (Connection Pooling) is mandatory to prevent Postgres from crashing under thousands of simultaneous connections.  
* Caching & Rate Limiting: Redis Cluster (AWS ElastiCache).  
  * Heavy reliance: Cache all eZee inventory reads for a few seconds. Do not query eZee at 500 RPS.  
* Payment & Commodity Gateway:  
  * The PWA queries our DB for commodity prices (Water Bottle \= ₹50). Vibe House uses Razorpay/Stripe to collect payment directly.  
  * Only after Vibe House's payment webhook confirms the payment, do we use a background job to post the charge to eZee's folios.

**D. Asynchronous Workers & Message Queues**

* Tech Stack: Redis \+ BullMQ (Node.js) or Kafka/RabbitMQ.  
* Purpose: At 500 RPS, any slow 3rd-party API call (eZee, MyGate) must be completely decoupled from the HTTP request cycle.  
  * Guest requests an item \-\> Queue job \-\> Send HTTP 202 Accepted to Guest instantly.  
  * Worker processes the queue safely based on the rate limits of eZee and MyGate.

**3\. Core System Flows**

**A. The Booking & Check-in Flow**

1. Booking: Guest books via OTA. eZee PMS receives the booking.  
2. Background Sync: Vibe House Workers poll/receive webhooks from eZee and create guest entries in PostgreSQL.  
3. Web Check-in: Guest authenticates and uploads KYC via PWA.  
4. Automation Trigger:  
* Operations pushed to Queue.  
* Worker 1: Calls eZee API Mark as Checked-In.  
* Worker 2: Calls MyGate API Create PIN (Valid until Checkout Time).

**B. The Commodity & Payment Flow (Inventory synced with eZee)**

Addressing the specific requirement: Custom Payment Gateway \+ eZee Inventory

1. **Catalog Load:** PWA fetches the catalog/prices from the Vibe House PostgreSQL DB (cached in Redis for extreme speed).  
2. Purchase Intent: Guest selects "1x Toilet Kit" (₹150).  
3. **Payment:** PWA initiates a payment through Razorpay/PhonePe.  
4. Webhook Confirmation: Razorpay sends a payment.captured webhook to our backend.  
5. **Atomic Transaction:**  
   * Backend verifies the webhook signature.  
   * Backend deducts 1 from the Vibe House inventory table.  
   * Backend creates an operational task for Housekeeping to deliver the kit.  
6. PMS Ledger Sync (Background): A queue worker safely calls the eZee AddExtraCharge API and the eZee Inventory API to deduct the stock over there, preventing any race conditions or 503 errors from eZee.

**4\. Addressing Infrastructure Reliability**

* Deployments without Downtime: AWS CodePipeline triggering ECS Rolling Updates. Old containers remain alive until new containers are confirmed healthy and serving traffic.  
* eZee / MyGate Rate Limits: BullMQ processes the background queues at a controlled pace (e.g., max 5 requests per second to eZee), even if Vibe House is receiving 500 RPS.  
* Database Load: Read replicas for PostgreSQL to offload heavy reporting queries from the Staff Dashboard.

**5\. Summary of the Essential Tech Stack**

* Frontend: Next.js (PWA)  
* Backend: Node.js (Fastify) or Go  
* Database: PostgreSQL (with PgBouncer)  
* Cache/Queue: Redis Cluster / BullMQ  
* Integrations: eZee PMS, MyGate (Smart Locks), Payment Gateway (Razorpay).  
* Infrastructure: AWS ECS/EKS (Dockerized), AWS ALB (Load Balancing).

**Tech Proposal**

**Why Async Workers & Message Queues?**

The Core Problem: At 500 RPS, you're constantly calling external APIs (eZee, MyGate, Razorpay). These calls are slow (100–2000ms each) and can fail. If you handle them directly in your HTTP request cycle, your guests wait for them all to finish, and one failure takes the whole request down.

---

**🔴 Without a Queue (Synchronous Flow)**

**Problem Statement: A guest buys a towel at ₹150 in the PWA.**

| Guest clicks "Buy Towel"       |Backend: verify payment (Razorpay API call) → 400ms       |Backend: deduct inventory in Postgres → 10ms       |Backend: POST to eZee AddExtraCharge API → 800ms       |Backend: POST to MyGate "create housekeeping task" → 600ms       |Backend: send WhatsApp message to guest → 500ms       |Total response time: \~2.3 seconds |
| :---- |

**What goes wrong:**

* eZee API is down at 11 PM → the entire purchase fails, guest keeps retrying, charges might duplicate  
* 50 guests buy something simultaneously → you fire 50 concurrent eZee API calls, eZee rate-limits you with 429 Too Many Requests, and you get chaos  
* Your backend thread is blocked waiting on slow APIs — at 500 RPS, your servers choke and queue traffic piles up

---

**With a Queue — Kafka or BullMQ**

**Same Problem Statement: Guest buys a towel.**

| Guest clicks "Buy Towel"       |Backend: verify payment (Razorpay API) → 400ms       |Backend: deduct inventory in Postgres → 10ms     ← ATOMIC. Done.       |Backend: push ONE message to Kafka topic:        { "event": "TOWEL\_PURCHASED", "guestId": 42, "roomId": 101, charge: 150 }    → 2ms       |⚡ Return HTTP 200 to guest in \~420ms total           \[Meanwhile, in the background...\]                        |           Kafka Consumer 1: POST to eZee AddExtraCharge           Kafka Consumer 2: Notify housekeeping dashboard           Kafka Consumer 3: Send WhatsApp to guest           (All at controlled pace, retrying on failure, no guest waiting) |
| :---- |

**What changes:**

| Scenario | Without Queue | With Kafka |
| :---- | :---- | :---- |
| eZee is down at 11 PM | Purchase fails entirely | Message sits in Kafka, replayed when eZee recovers |
| 50 guests buy simultaneously | 50 parallel eZee API calls (rate-limit crash) | Kafka consumer processes them at controlled pace (e.g., 5/second) |
| WhatsApp gateway is slow | Guest waits 2+ seconds | Guest gets instant 200 OK, WhatsApp sends eventually |
| Server crashes mid-flow | Data inconsistency | Kafka message persisted on disk, processed after restart |

# **Guest Booking Flow — DB Walkthrough & PRIMARY Role Assignment**

## **Question 1: Guest Booking Flow in the Database**

**Prompt:** *I need to know a question, a guest vs one who has actually booked a bed. Say for example ABC has booked a bed from MMT, but he hasn’t signed in as a guest yet, he got the booking\_id (eZee reservation ID) from a medium and now he logs in to our site, how will his flow go in terms of DB. Can you please answer this.*

**Answer:**

Great question — let me walk you through exactly what happens in the DB, step by step.

### **The Flow: ABC books on MMT → Gets eZee reservation ID → Signs up on Vibe House**

**Step 1: Booking happens on MMT (Before Vibe House knows anything)**

At this point, nothing exists in our database. The booking lives entirely in eZee PMS because MMT pushed it there via eZee’s channel manager. Our DB has zero rows for ABC.

| Our DB:       \[empty\]eZee PMS:     Reservation \#RES-12345 → ABC, Room Dorm-LB, Mar 15-18 |
| :---- |

**Step 2: ABC opens the Vibe House PWA and signs up**

ABC signs up via Google OAuth or phone OTP. Now our DB has:

| \-- guests table: 1 row createdINSERT INTO guests (id, name, email, phone, email\_verified)VALUES ('guest-001', 'ABC', 'abc@gmail.com', '+91-9876543210', true);\-- auth\_providers table: 1 row createdINSERT INTO auth\_providers (id, guest\_id, provider, provider\_uid)VALUES ('ap-001', 'guest-001', 'google', 'google-sub-id-xyz'); |
| :---- |

At this point ABC is a “guest” — but has NO booking linked. He’s just an authenticated user with no services unlocked.

**Step 3: ABC enters the eZee reservation ID on the PWA**

ABC clicks “Link Booking” and enters RES-12345 (received via MMT confirmation email/SMS).

What the backend does:

| \-- 1\. Call eZee API: GET reservation details for RES-12345\--    Response: { room\_type: "Dorm-LB", room: "101", bed: "101-A",\--                checkin: "2026-03-15", checkout: "2026-03-18", source: "MMT" }\-- 2\. ezee\_booking\_cache: 1 row created (cache the booking locally)INSERT INTO ezee\_booking\_cache  (ezee\_reservation\_id, property\_id, guest\_id, room\_type\_name, room\_number,   unit\_code, checkin\_date, checkout\_date, source, status, fetched\_at)VALUES  ('RES-12345', 'prop-001', 'guest-001', 'Dorm-LB', '101',   '101-A', '2026-03-15', '2026-03-18', 'MMT', 'RESERVED', NOW());\-- 3\. booking\_guest\_access: 1 row \-- ABC is PRIMARY, auto-approvedINSERT INTO booking\_guest\_access  (id, ezee\_reservation\_id, guest\_id, role, status, approved\_at)VALUES  ('bga-001', 'RES-12345', 'guest-001', 'PRIMARY', 'APPROVED', NOW()); |
| :---- |

Now ABC is a “guest with a booking”. The KYC, add-ons, and all other services are unlocked.

**Step 4: ABC’s friend DEF joins later**

DEF signs up separately → enters the same RES-12345 → system detects it’s already linked to ABC (PRIMARY) → sends OTP to ABC for approval:

| \-- 1\. Call eZee API: GET reservation details for RES-12345\--    Response: { room\_type: "Dorm-LB", room: "101", bed: "101-A",\--                checkin: "2026-03-15", checkout: "2026-03-18", source: "MMT" }\-- 2\. ezee\_booking\_cache: 1 row created (cache the booking locally)INSERT INTO ezee\_booking\_cache  (ezee\_reservation\_id, property\_id, guest\_id, room\_type\_name, room\_number,   unit\_code, checkin\_date, checkout\_date, source, status, fetched\_at)VALUES  ('RES-12345', 'prop-001', 'guest-001', 'Dorm-LB', '101',   '101-A', '2026-03-15', '2026-03-18', 'MMT', 'RESERVED', NOW());\-- 3\. booking\_guest\_access: 1 row \-- ABC is PRIMARY, auto-approvedINSERT INTO booking\_guest\_access  (id, ezee\_reservation\_id, guest\_id, role, status, approved\_at)VALUES  ('bga-001', 'RES-12345', 'guest-001', 'PRIMARY', 'APPROVED', NOW()); |
| :---- |

**Visual Summary**

| Timeline | guests | ezee\_booking\_cache | booking\_guest\_access |
| :---- | :---- | :---- | :---- |
| MMT books on eZee | — | — | — |
| ABC signs up on PWA | ✅ 1 row | — | — |
| ABC enters RES-12345 | (same) | ✅ 1 row cached | ✅ PRIMARY / APPROVED |
| DEF signs up | ✅ 2 rows | (same) | (same) |

The key insight: A guest row alone gives you nothing but a login. You can only access services (KYC, add-ons, lock PIN) when you have an APPROVED row in booking\_guest\_access linking you to an ezee\_booking\_cache entry. That’s the gate.

## **Question 2: What if DEF Enters the Booking ID Before ABC?**

**Prompt:** *What if DEF enters and logins first with booking ID whereas ABC has booked the entire booking from MMT? Ideally ABC should be the primary user? Is there a way we are matching the email id of primary user in eZee (provided during reservation by ABC), so that even if DEF was the first to signup and enter the booking id in our website, we still give him a secondary?*

**Answer:**

Sharp edge case — and yes, there’s a clean way to solve it using eZee’s reservation data. Here’s the problem and the fix:

### **The Problem**

When DEF enters RES-12345 first, our current logic would blindly make whoever enters first \= PRIMARY. That’s wrong. ABC booked it on MMT, so eZee has ABC’s contact details attached to the reservation.

### **The Fix: Match Against eZee Booker Contact Info**

When eZee created the reservation from MMT, it stores the booker’s name, email, and phone on the reservation record. When we call eZee’s API to populate ezee\_booking\_cache, we fetch and store that too:

**Schema Update — Add 2 columns to ezee\_booking\_cache**

| booker\_email  VARCHAR(255)   \-- ABC's email as registered on MMT/eZeebooker\_phone  VARCHAR(20)    \-- ABC's phone as registered on MMT/eZee |
| :---- |

### **The Revised Role Assignment Logic**

When any guest enters a booking ID, the backend runs this check:

| Guest enters RES-12345    ↓Call eZee API → fetch reservation    ↓Cache to ezee\_booking\_cache (including booker\_email \+ booker\_phone)    ↓Does the logged-in guest's email OR phone  match booker\_email OR booker\_phone?    ↓ YES                           ↓ NOrole \= PRIMARY                  role \= SECONDARYstatus \= APPROVED               status \= PENDING\_APPROVAL(no OTP needed)                 (send OTP to PRIMARY to approve) |
| :---- |

### **Walking Through the Scenario**

**Scenario: DEF logs in first and enters RES-12345. ABC hasn’t signed up yet.**

**DEF logs in:**

| guests row → { id: 'guest-DEF', email: 'def@gmail.com', phone: '+91-1111' } |
| :---- |

**DEF enters RES-12345:**

| eZee returns → { ..., booker\_email: 'abc@gmail.com', booker\_phone: '+91-9999' } |
| :---- |

**Cache stored:**

| ezee\_booking\_cache → { ezee\_reservation\_id: 'RES-12345',                        booker\_email: 'abc@gmail.com',                        booker\_phone: '+91-9999' } |
| :---- |

**Match check:**

| def@gmail.com \== abc@gmail.com?  ❌\+91-1111 \== \+91-9999?            ❌ |
| :---- |

→ DEF gets: role \= SECONDARY, status \= PENDING\_APPROVAL

→ Send OTP to abc@gmail.com / \+91-9999: “Someone wants access to your booking. Approve?”

**Later, ABC signs up and enters RES-12345:**

| ABC logs in:  guests row → { id: 'guest-ABC', email: 'abc@gmail.com' }Match check:  abc@gmail.com \== booker\_email?   ✅ |
| :---- |

→ ABC gets: role \= PRIMARY, status \= APPROVED (immediately)

→ ABC can now see DEF’s PENDING\_APPROVAL and approve/reject it

### **What if ABC never signs up on Vibe House?**

This is the tricky case. DEF is stuck as PENDING\_APPROVAL forever waiting for ABC to approve. Two options:

**Option A — Staff override:**

Front desk can manually approve DEF from the Zoho dashboard when DEF shows up at the property with a valid ID.

**Option B — Time-based auto-approve:**

If check-in date passes and ABC still hasn’t signed up, auto-promote DEF to PRIMARY. (This is an ops decision for you to make — I’d recommend Option A for security.)

### **Summary of How PRIMARY Is Determined**

The key is fetching booker\_email and booker\_phone from eZee when the reservation is first cached, then matching against the authenticated guest’s details:

| Guest enters booking ID        ↓Fetch from eZee API → cache in ezee\_booking\_cache  (includes booker\_email, booker\_phone from the OTA reservation)        ↓Compare:  guest.email \== booker\_email?  OR  guest.phone \== booker\_phone?        ↓  YES → PRIMARY \+ APPROVED immediately (no OTP needed)  NO  → SECONDARY \+ PENDING APPROVAL (OTP sent to booker's email/phone) |
| :---- |

### **Edge Cases Now Handled**

| Scenario | Outcome |
| :---- | :---- |
| ABC (booker) signs up first, enters booking ID | ✅ PRIMARY — email matches |
| DEF enters booking ID before ABC | ✅ SECONDARY — no email match, wait for ABC’s OTP approval |
| ABC’s email on MMT was different from PWA email | ⚠️ Falls back to phone match. If both differ, DEF gets SECONDARY — staff can manually override via Zoho |
| ABC never signs up on Vibe House before check-in | ⚠️ DEF stays PENDING. Front desk promotes manually via Zoho at the kiosk |

The two new columns — booker\_email and booker\_phone — are your source of truth for this matching logic. No additional tables needed.
